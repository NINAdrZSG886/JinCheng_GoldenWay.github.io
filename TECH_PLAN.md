# 通用静态网页上位机技术规划（Edge / Web Bluetooth / Web Serial）

## 1. 项目目标与原则

### 1.1 核心目标
构建一个**纯静态网页**的通用上位机框架，用户通过 Edge 浏览器打开页面即可连接设备并进行交互。

### 1.2 不变原则
1. **无后台依赖**：系统必须可部署在 GitHub Pages 或本地静态资源环境中运行。
2. **强配置驱动**：用户优先通过配置文件完成页面和协议改造，尽量避免改动源码。
3. **调试优先**：运行状态、协议行为、配置错误必须可观测、可定位、可导出。
4. **轻量可维护**：保持最小可行体积和清晰架构，避免引入重量级运行时。

---

## 2. 总体架构设计

系统采用“配置层 + 引擎层 + 插件层”三层结构：

### 2.1 配置层（Config Layer）
负责描述页面布局、协议参数、数据绑定关系。

- `config/app.json`：应用级参数（页面列表、主题、日志等级、默认连接策略）
- `config/pages/*.json`：页面布局和组件树
- `config/protocols/*.json`：协议模板参数（帧格式、命令定义、超时重试）
- `config/bindings/*.json`：组件与协议点位映射

### 2.2 引擎层（Engine Layer）
负责运行时执行与调度。

- `ui-engine`：根据页面配置动态渲染组件
- `protocol-engine`：统一处理协议流程（主动查询、回复模式、定时任务、条件触发）
- `transport`：抽象通讯通道（Web Bluetooth / Web Serial）
- `validator`：启动时校验配置合法性与引用关系
- `logger`：结构化日志与运行状态上报

### 2.3 插件层（Plugin Layer）
提供可扩展能力。

- 组件插件（按钮、开关、数值显示、曲线图等）
- 协议插件（自定义编码/解码/校验逻辑）
- DSL行为插件（受控脚本逻辑）

---

## 3. 协议系统设计（重点）

## 3.1 通用协议行为模型
协议行为抽象为可配置任务，支持以下模式：

1. **主动查询模式（Polling）**
   - 定时发送查询帧
   - 解析响应帧
   - 更新绑定数据点

2. **回复模式（Responder）**
   - 接收请求帧
   - 按匹配规则触发响应
   - 发送对应回复帧

3. **条件查询模式（Conditional Task）**
   - 当条件成立（例如某点位变化/阈值超限）触发查询或控制命令

4. **定时任务模式（Scheduler）**
   - 支持固定周期、延迟执行、有限重试、失败回退策略

### 3.2 帧格式可配置能力
用户可通过配置定义协议帧结构：

- 帧头/帧尾
- 长度字段位置
- 命令字与地址映射
- 校验方式（例如 CRC16/XOR）
- 字节序（LE/BE）
- 分包与重组规则

### 3.3 协议配置示例（抽象）
配置应支持：

- `commands`：命令模板
- `timers`：轮询/定时计划
- `matchers`：帧匹配规则
- `actions`：匹配后的动作（更新点位/触发发送/告警）
- `retries`：超时重试与熔断策略

## 4. Lua风格 DSL 设计（不引入 Lua 运行时）

## 4.1 设计目标
在不引入 Lua VM 的前提下，提供“类似 Lua 的脚本表达力”，用于高级协议逻辑（特别是 OTA 升级流程）。

### 4.2 约束原则
1. **受限语法**：仅支持白名单语法与内置函数。
2. **静态编译**：DSL 在加载时编译为受控中间表示（IR）或受控 JS 函数。
3. **禁止动态执行**：不允许 `eval` / 动态网络加载脚本。
4. **可审计可回溯**：每条 DSL 规则可映射回配置源位置。

### 4.3 计划支持的基础语法能力
- 变量定义与赋值
- `if / elseif / else`
- 布尔表达式与比较运算
- 基本循环（可选，需限制次数）
- 调用内置协议函数（发送、等待、解析、校验、写点）

### 4.4 OTA 场景建议内置函数
- `send_frame(cmd, payload)`
- `wait_frame(matcher, timeout_ms)`
- `verify_crc(data)`
- `split_firmware(chunk_size)`
- `report_progress(percent)`
- `abort(reason)`

## 5. 页面系统设计

页面由配置定义，支持多页面与网格布局。

每个组件建议包含：
- `id`：唯一标识
- `type`：组件类型
- `layout`：位置与尺寸（x, y, w, h）
- `protocolRef`：关联协议
- `dataSource`：绑定数据点
- `props`：显示配置
- `actions`：交互动作（点击后发送命令等）

页面渲染流程：
1. 读取页面配置
2. 校验组件类型与属性
3. 渲染组件
4. 建立数据绑定
5. 注册动作与生命周期

---

## 6. 调试与可观测性设计

调试系统必须作为一等公民实现。

### 6.1 右下角日志面板（Dock）
- 日志等级：`debug/info/warn/error`
- 日志分类：`config/ui/protocol/transport/runtime`
- 能力：筛选、搜索、清空、导出

### 6.2 运行状态面板（Health Board）
- 组件加载状态（✅/❌）
- 协议任务状态（运行中/暂停/失败）
- 通讯状态（已连接/断开/重连中）
- 最近错误摘要

### 6.3 启动自检（Boot Diagnostics）
启动阶段自动检查：
1. 配置文件完整性
2. Schema 合法性
3. 引用关系（页面->组件->协议->绑定）
4. 传输能力可用性（浏览器 API 支持）

输出统一诊断报告，支持下载。

---

## 7. 安全与兼容性边界

1. 优先支持 Edge 最新稳定版。
2. 依赖浏览器安全上下文（HTTPS 或 localhost）。
3. 对 Web Bluetooth / Web Serial 权限行为做显式提示。
4. DSL 执行必须设置超时与资源上限，防止死循环或阻塞。

## 9. 分阶段实施路线图

### Phase 1：最小可运行框架
- 建立配置加载、校验、日志系统
- 实现基础页面渲染与 1~2 个组件
- 打通蓝牙连接与基础收发

### Phase 2：协议引擎可配置化
- 完成主动查询、回复模式、定时任务
- 帧格式参数化（帧头帧尾、校验、重试）
- 数据绑定闭环

### Phase 3：DSL 能力上线
- 实现 Lua风格 DSL 语法子集
- 完成静态编译与安全执行器
- 提供 OTA 脚本模板与示例

### Phase 4：插件化与通用化
- 发布组件插件 API
- 发布协议插件 API
- 发布标准模板库（不同设备类型）

### Phase 5：工程稳定化
- 配置版本迁移机制
- 诊断报告完善
- 文档与示例工程完善

---

## 10. 成功标准（验收指标）

1. 新用户可在不改核心源码前提下，通过修改 `config` 完成基础上位机搭建。
2. 协议行为可通过配置 + DSL 完成大部分业务逻辑。
3. 运行中的配置错误与协议错误可在 3 分钟内定位。
4. 整体仍保持纯静态部署，无后端依赖。
